# OS 文件系统项目详解

> 一个基于 inode 的模拟文件系统，C 语言实现后端，PyQt6 实现可视化前端。支持完整的文件系统操作，并通过 GUI 实时展示内部状态。

---

## 一、项目概述

本项目是一个教学用的模拟文件系统，从 16 位 Turbo C 原始代码移植到现代 32/64 位系统。核心目标是在单个磁盘镜像文件（`disk.img`）上实现一个类 Unix 的文件系统，并通过可视化前端让用户直观理解文件系统的内部工作机制。

**核心特性：**
- 基于 inode 的文件管理（512 个 inode，512 个数据块）
- 多级索引支持大文件（直接索引 + 一级间接索引）
- 完整的 UGO 权限模型
- 硬链接与软链接
- 目录树结构
- TCP Daemon 模式，支持远程/本地访问
- PyQt6 可视化前端，实时展示磁盘块占用、inode 分配、目录结构
- 块级依赖图、历史状态回放等高级可视化功能

## 二、文件系统磁盘架构

### 2.1 磁盘布局

整个文件系统存储在单个文件 `disk.img` 中，总大小约 280KB。磁盘按 512 字节/块划分：

| 区域 | 块号 | 大小 | 说明 |
|------|------|------|------|
| 超级块 | 0 | 512B | 文件系统全局元数据 |
| 系统保留 | 1 | 512B | 引导块等保留 |
| inode 区 | 2–33 | 32 × 512B = 16KB | 512 个 inode，每个 32B |
| 数据区 | 34–545 | 512 × 512B = 256KB | 用户数据块 |

**关键宏定义（filesys.h）：**
```c
#define BLOCKSIZ    512     // 块大小
#define DINODEBLK   32      // inode 占用的块数
#define FILEBLK     512     // 数据块总数
#define DINODESIZ   32      // 磁盘 inode 大小
#define DIRSIZ      14      // 文件名最大长度
#define NADDR       10      // inode 地址数组长度
```

### 2.2 超级块（Super Block）

超级块存储在磁盘 0# 块，是文件系统的"大脑"，记录全局状态：

```c
struct filsys {
    uint16_t s_isize;       // inode 区占用的块数
    uint16_t s_fsize;       // 文件系统总块数
    uint16_t s_nfree;       // 当前空闲块数量
    uint16_t s_free[50];    // 空闲块栈（成组链接法）
    uint16_t s_ninode;      // 当前空闲 inode 数量
    uint16_t s_inode[50];   // 空闲 inode 数组
    uint16_t s_fmod;        // 超级块是否被修改（脏标记）
    uint16_t s_rinode;      // 空闲 inode 搜索起点
    uint16_t s_pinode;      // 空闲 inode 栈指针
};
```

超级块在内存中有一份拷贝（全局变量 `filsys`），修改后通过 `sb_write()` 写回磁盘。

### 2.3 磁盘 inode 结构（dinode）

每个文件/目录对应一个 inode，磁盘上固定 32 字节：

```c
struct dinode {
    uint16_t di_number;     // 链接计数（硬链接数）
    uint16_t di_mode;       // 文件类型 + 权限
    uint16_t di_uid;        // 所有者用户 ID
    uint16_t di_gid;        // 所属组 ID
    uint32_t di_size;       // 文件大小（字节）
    uint16_t di_addr[10];   // 数据块地址数组
    uint32_t di_ctime;      // 创建时间
};
```

**di_mode 编码：**
- 高 4 位：文件类型（`DIFILE=01000` 普通文件、`DIDIR=02000` 目录、`DILINK=04000` 链接）
- 低 9 位：UGO 权限（用户/组/其他各 rwx 3 位）

### 2.4 内存 inode 结构

磁盘 inode 读入内存后，会扩展为更丰富的内存结构：

```c
struct inode {
    struct inode *i_forw;   // Hash 链表前向指针
    struct inode *i_back;   // Hash 链表后向指针
    char i_flag;            // 状态标记（IUPDATE 等）
    uint16_t i_ino;         // inode 号
    uint16_t i_count;       // 引用计数（内存中多少处引用）
    uint16_t di_number;     // 链接计数
    uint16_t di_mode;
    uint16_t di_uid;
    uint16_t di_gid;
    uint32_t di_size;
    uint16_t di_addr[10];
};
```

内存 inode 使用 **Hash 链表**管理（128 个桶），加速查找。`iget()` 先在 Hash 链表中查找，未命中才从磁盘读取。

### 2.5 目录项结构

目录文件的内容由一系列 `direct` 结构组成，每个 16 字节：

```c
struct direct {
    char d_name[14];        // 文件名
    uint16_t d_ino;         // 对应的 inode 号
};
```

一个 512B 的数据块可存储 32 个目录项。

### 2.6 多级索引机制

`di_addr[10]` 的索引策略：

| 索引 | 用途 | 寻址范围 |
|------|------|----------|
| di_addr[0]–di_addr[8] | 直接索引 | 9 × 512B = 4.5KB |
| di_addr[9] | 一级间接索引块 | 128 × 512B = 64KB（总上限约 70KB）|

**大文件写入流程：**
1. 前 9 个逻辑块 → 直接存入 di_addr[0]–di_addr[8]
2. 第 10 个及以后的逻辑块 → di_addr[9] 指向一个间接索引块，该块中存储 128 个 uint16_t 的物理块号

```
小文件（≤9 块）：
  inode.di_addr[0] = blk_100
  inode.di_addr[1] = blk_101
  ...

大文件（>9 块）：
  inode.di_addr[0..8] = 直接数据块
  inode.di_addr[9] = blk_indirect（间接索引块）
  blk_indirect 内容：[blk_200, blk_201, blk_202, ...]（128 个块号）
```

### 2.7 空闲块管理：成组链接法

文件系统使用**成组链接法（Group Linking）**管理空闲数据块，这是类 Unix 系统的经典算法。它将 512 个空闲数据块分成若干组，用"超级块栈 + 磁盘组长块链"两级结构管理，兼顾了分配回收的速度和磁盘空间的利用率。

---

#### 核心数据结构

成组链接法的全部状态存储在**超级块**中：

```c
struct filsys {
    uint16_t s_nfree;       // 当前手边有多少个空闲块
    uint16_t s_free[50];    // 手边的空闲块编号栈（NICFREE = 50）
    uint16_t s_pfree;       // 栈指针：指向栈顶元素的下标
    ...
};
```

**关键规则：**
- 栈最多存 50 个块号（`NICFREE`）
- **栈底**（`s_pfree == 49`）的那个块号是**组长块**，它的数据内容存着下一组的块号
- 分配时从栈顶弹出，回收时压入栈顶

---

#### 分配流程：balloc()

**场景："给我一个空闲块"**

假设当前超级块栈中有 50 个块号：

```
s_free = [100, 101, 102, ..., 149]   ← 100 是栈顶（s_pfree = 0）
                                    ← 149 是栈底（s_pfree = 49）
s_nfree = 50
```

**前 49 次分配（正常弹出）：**
```
第 1 次: 弹出 100，s_pfree = 1，返回 100
第 2 次: 弹出 101，s_pfree = 2，返回 101
...
第 49 次: 弹出 148，s_pfree = 49，返回 148
```

**第 50 次分配（遇到组长块）：**
```
s_pfree = 49，s_free[49] = 149
发现是栈底！149 不是普通数据块，它是"组长块"

1. 读取 149 号块的内容（512 字节）:
   第 0~1 字节:  count = 50（下一组有多少个）
   第 2~3 字节:  200（第 1 个块号）
   第 4~5 字节:  201（第 2 个块号）
   ...
   第 100~101 字节: 249（第 50 个块号）

2. 把这 50 个块号反向装入 s_free 栈:
   s_free = [200, 201, 202, ..., 249]
   s_pfree = 0（新的栈顶指向 200）

3. 返回 149（组长块本身也被分配出去，当作普通数据块用）
```

**代码（ballfre.c）：**
```c
uint16_t balloc(void)
{
    uint16_t free_block = filsys.s_free[filsys.s_pfree];

    if (filsys.s_pfree == NICFREE - 1) {
        // 栈底是组长块，读取下一组
        fseek(fd, DATASTART + free_block * BLOCKSIZ, SEEK_SET);
        fread(block_buf, 1, BLOCKSIZ, fd);
        uint16_t next_count = block_buf[0];  // 第一字是计数
        for (int i = 0; i < next_count; i++) {
            filsys.s_free[NICFREE - 1 - i] = block_buf[i + 1];
        }
        filsys.s_pfree = NICFREE - next_count;
    } else {
        filsys.s_pfree++;
    }

    filsys.s_nfree--;
    filsys.s_fmod = SUPDATE;  // 标记超级块为脏
    return free_block;
}
```

---

#### 回收流程：bfree()

**场景："把这个块还回来"**

假设当前栈中有 49 个块号，还能装 1 个：

```
s_free = [200, 201, ..., 248]   ← 49 个块
s_pfree = 0
s_nfree = 49
```

**前 1 次回收（正常压入）：**
```
回收 300:
s_pfree 不需要变化（因为是从 0 开始压入）
s_free[0] = 300
s_nfree = 50
```

**第 2 次回收（栈满，创建新组长块）：**
```
回收 301:
栈已经满了（50 个）！

1. 把当前这 50 个块号写入 301 号块:
   第 0~1 字节:  count = 49（注意：不包含组长块自己）
   第 2~3 字节:  300
   第 4~5 字节:  248
   ...
   （反向存储，因为 balloc 读取时会再反向装回）

2. 301 号块现在成为新的组长块

3. 超级块栈重置:
   s_free[49] = 301
   s_pfree = 49
   s_nfree = 1
```

**代码（ballfre.c）：**
```c
void bfree(uint16_t block_num)
{
    if (block_num < 3) return;  // 系统保留块不回收

    if (filsys.s_pfree == 0) {
        // 栈满，把当前栈写入回收块，成为新组长块
        block_buf[0] = NICFREE - 1;  // 计数 49
        for (int i = 0; i < NICFREE - 1; i++) {
            block_buf[i + 1] = filsys.s_free[NICFREE - 1 - i];
        }
        fseek(fd, DATASTART + block_num * BLOCKSIZ, SEEK_SET);
        fwrite(block_buf, 1, BLOCKSIZ, fd);
        filsys.s_pfree = NICFREE - 1;
    } else {
        filsys.s_pfree--;
    }

    filsys.s_free[filsys.s_pfree] = block_num;
    filsys.s_nfree++;
    filsys.s_fmod = SUPDATE;
}
```

#### 格式化时的初始化

`format` 时所有数据块都是空闲的，`sb_init_format()`（superblock.c）负责建立初始的组长块链：

**步骤：**
1. 把 509 个空闲块（3~511）按**倒序**排成一条序列：
   ```
   [511, 510, 509, ..., 3]
   ```

2. 前 50 个（511~462）直接放入超级块栈：
   ```
   s_free = [511, 510, ..., 462]
   s_pfree = 0
   s_nfree = 509
   ```

3. 从第 50 个开始，每 50 个为一组，**最后一个块作为组长块**：
   ```
   第 1 组: 511~462（在超级块栈中）
   组长块 461 的内容: [49, 460, 459, ..., 412]  ← 计数 + 49 个块号
   
   第 2 组: 460~412（在组长块 461 中）
   组长块 411 的内容: [49, 410, 409, ..., 362]
   
   ...
   
   最后一组可能不足 50 个，特殊处理
   ```

4. 为什么要**反向**存储？
   - `balloc` 读取组长块时，会把块号**反向**装入 `s_free` 栈
   - 这样两次反向就抵消了，保证栈顶始终是逻辑上"下一个要分配的块"

**完整的链式结构示意：**
```
超级块栈（内存）
├─ 511, 510, ..., 462  ← 直接分配
└─ 461 是组长块 → 读取内容得到下一组

461 号块（磁盘）
├─ 460, 459, ..., 412  ← 分配
└─ 411 是组长块 → 读取内容得到下一组

411 号块（磁盘）
├─ 410, 409, ..., 362  ← 分配
└─ 361 是组长块 → ...

...一直链到最后...
```

---

#### 为什么用成组链接法？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **位图** | 简单，O(1) 查找 | 512 个位 = 64 字节，但找连续空闲块慢 |
| **链表** | 分配回收都 O(1) | 每个块存一个 next 指针，浪费 2 字节/块 |
| **成组链接法** | 分配回收 O(1)，不浪费空间 | 实现稍复杂 |

成组链接法的精髓：**用超级块缓存一批块号，避免每次分配都读磁盘**。只有在缓存用完时才读一次组长块，把 I/O 批量化了。对于 512 个块的小型文件系统，几乎所有分配操作都在内存中完成。

### 2.8 空闲 inode 管理

空闲 inode 采用更简单的**栈式管理**：
- 超级块 `s_inode[50]` 缓存最多 50 个空闲 inode 号
- `s_pinode` 是栈顶指针，`s_ninode` 是栈中数量
- 分配时从栈顶弹出，回收时压入栈顶
- 栈空时从 `s_rinode` 开始向后扫描磁盘 inode 位图补充

## 三、后端源码文件详解

### 3.1 main.c（主程序 / daemon）

这是整个后端的入口，包含两个核心功能：

**A. TCP Daemon 模式**

```c
run_tcp_daemon(host, port)
```
- 绑定 TCP Socket，监听 `127.0.0.1:9090`
- `accept()` 循环，一次只处理一个客户端连接
- `handle_client()`：从 socket 读取命令行，调用 `handle_daemon_command()`
- `handle_daemon_command()`：用 `dup2` 将 stdout 重定向到 tmpfile，执行命令，再把 tmpfile 内容通过 socket 回传，最后追加 `__END__\n`

**B. 命令分发（process_cmd）**

解析用户输入的命令字符串，分发给各个子模块：
- `login` → `cmd_login`
- `dir` / `ls` → `cmd_dir`
- `create` → `cmd_create`
- `open` → `cmd_open`
- `close` → `cmd_close`
- `read` / `write` → `read()` / `write()`（系统调用级函数）
- `delete` / `rm` → `cmd_delete`
- `mkdir` → `cmd_mkdir`
- `rmdir` → `cmd_rmdir`
- `cd` → `cmd_cd`
- `pwd` → `cmd_pwd`
- `cp` → `cmd_cp`
- `mv` → `cmd_mv`
- `ln` → `cmd_ln`
- `symlink` → `cmd_symlink`
- `chmod` → `cmd_chmod`
- `chown` → `cmd_chown`
- `find` → `cmd_find`
- `stat` → `cmd_stat`
- `wc` → `cmd_wc`
- `df` → `cmd_df`
- `lseek` → `cmd_lseek`
- `format` → `cmd_format`
- `help` → `show_help`

**C. 内部查询命令**

专供前端刷新面板使用的无颜色输出命令：
- `__disk_map__`：输出 512 个块的占用状态（0/1 空格分隔）
- `__inode_list__`：输出所有已分配 inode 的表格（`ino|type|size|mode|uid|gid|links|blocks`）
- `__dir_list__`：输出当前目录的文件列表（`name|ino|type|size|mode`）
- `__tree__`：输出目录树的缩进文本
- `__stats__`：输出系统统计（`key=value` 格式）
- `__block_info__`：输出块与 inode 的归属映射（`block|ino|role|index`）
- `__snapshot__`：**v5.0 新增**，一次返回上述所有信息（磁盘块 + inode 列表 + 目录 + 目录树 + 统计 + 块归属），用 `__SECTION__ name` 分隔各段，减少 socket 往返次数

**跨平台修复：** 原始代码的 `read`/`write`/`close` 函数名与系统调用冲突，在 `handle_daemon_command` 中使用 `syscall(SYS_read/write/close)` 直接调用系统调用。

### 3.2 filesys.h（头文件）

整个项目的核心头文件，定义了所有数据结构、常量、全局变量和函数声明。

**关键设计决策：**
- 使用 `#pragma pack(push, 1)` 禁止编译器填充，确保磁盘结构体大小精确可控
- 使用 `<stdint.h>` 的定宽类型（`uint16_t`、`uint32_t`）替代 `unsigned int`，解决 16 位/32 位/64 位系统之间的兼容性问题
- 补齐所有函数的 ANSI C 风格声明（返回类型 + 参数类型）

**全局变量：**
- `fd`：磁盘镜像文件的 FILE 指针
- `filsys`：内存中的超级块
- `hinode[128]`：inode Hash 表
- `sys_ofile[40]`：系统打开文件表
- `user[10]`：用户表
- `cur_path_inode`：当前目录的内存 inode 指针
- `dir`：当前目录的内存缓存

### 3.3 igetput.c（inode 读写与缓存）

**`iget(uint16_t dinodeid)` — 获取 inode**
1. 计算 Hash 桶：`inodeid % 128`
2. 在 Hash 链表中查找，命中则引用计数 `i_count++`，直接返回
3. 未命中则 `malloc` 分配新内存 inode，从磁盘读取 dinode 内容
4. 插入 Hash 链表头部

**`iput(struct inode *pinode)` — 释放 inode**
1. `i_count--`，如果仍被引用则直接返回
2. 如果 `di_number == 0`（无硬链接），释放所有数据块，标记为 DIEMPTY，写回磁盘，回收 inode
3. 如果 `di_number != 0`，写回磁盘（脏数据同步）
4. 从 Hash 链表中摘除，释放内存

这是文件系统的核心缓存层，避免重复的磁盘 I/O。

### 3.4 iallfre.c（inode 分配与回收）

**`ialloc()` — 分配 inode**
1. 如果超级块空闲 inode 栈 `s_inode` 非空，弹出栈顶
2. 如果栈空，从磁盘扫描 inode 区，将空闲 inode 号补充入栈（最多 50 个）
3. 初始化新 inode 的 mode、uid、gid、size、addr 等字段
4. 返回新 inode 号

**`ifree(uint16_t inodeid)` — 回收 inode**
1. 如果超级块栈未满，压入栈顶
2. 如果栈满，直接丢弃（inode 号不会丢失，下次 `ialloc` 扫描时会重新发现）

### 3.5 ballfre.c（数据块分配与回收）

成组链接法的实现，前文已详细描述。

**`balloc()` — 分配数据块**
- 从超级块空闲栈弹出块号
- 栈底时读取组长块补充栈

**`bfree(uint16_t block_num)` — 回收数据块**
- 压入超级块空闲栈
- 栈满时将当前栈写入新组长块

### 3.6 rdwt.c（文件读写 + 多级索引）

**`get_block(inode, logical_block)` — 逻辑块号 → 物理块号**
- 逻辑块 < 9：直接返回 `di_addr[logical_block]`
- 逻辑块 ≥ 9：读取间接索引块 `di_addr[9]`，从中取出物理块号

**`set_block(inode, logical_block, physical_block)` — 建立映射**
- 逻辑块 < 9：直接存入 `di_addr`
- 逻辑块 ≥ 9：如果间接块不存在，分配一个间接块，初始化后写入映射

**`read(fd_num, buf, size)` — 读文件**
1. 通过 fd 找到内存 inode 和当前偏移量
2. 计算当前偏移所在的逻辑块号和块内偏移
3. 调用 `get_block` 获取物理块号
4. `fseek` + `fread` 读取数据（跨块时循环）
5. 更新偏移量

**`write(fd_num, buf, size)` — 写文件**
1. 类似 `read`，但遇到未分配的逻辑块时调用 `balloc` 分配新块
2. 调用 `set_block` 建立映射
3. `fseek` + `fwrite` 写入数据
4. 如果写入超出原文件大小，更新 `di_size`
5. 标记 inode 为 IUPDATE（脏）

**`free_all_blocks(inode)` — 释放 inode 的所有数据块**
- 遍历直接索引块，逐个 `bfree`
- 如果有间接索引块，读取间接块内容，逐个释放指向的数据块，最后释放间接块本身
- 清空 `di_addr` 数组

### 3.7 creat.c（文件创建）

**`creat(user_id, filename, mode)` — 创建文件**
1. 检查文件名是否已存在（`namei` 查找）
2. 如果存在且是文件，截断（释放旧数据块，重置 size 和 addr）
3. 如果不存在，分配新 inode（`ialloc`）
4. 在当前目录中添加目录项（`d_name` + `d_ino`）
5. 设置文件类型为 `DIFILE`，权限为传入的 mode
6. 在系统打开文件表和用户 fd 表中分配一个空位
7. 返回 fd 号

### 3.8 open.c（文件打开）

**`aopen(user_id, filename, mode)` — 打开文件**
1. `namei` 查找文件名，获取 inode
2. 检查权限（`access`）
3. 在系统打开文件表和用户 fd 表中分配空位
4. 根据 mode 设置打开方式（FREAD / FWRITE / FAPPEND）
5. 如果是 append 模式，将文件偏移设为文件末尾
6. 返回 fd 号

### 3.9 close.c（文件关闭）

**`close(user_id, fd_num)` — 关闭文件**
1. 减少系统打开文件表的引用计数
2. 如果引用计数降为 0，释放系统打开文件表项
3. `iput` 写回 inode
4. 清除用户 fd 表项

### 3.10 delete.c（文件删除）

**`delete(user_id, filename)` — 删除文件**
1. `namei` 查找文件名，获取 inode
2. 检查是否为目录（拒绝删除目录）
3. 检查写权限
4. 从当前目录中删除目录项
5. `di_number--`（减少链接计数）
6. `iput`：如果链接计数降为 0，释放所有数据块并回收 inode

### 3.11 dir.c（目录操作）

**`mkdir(user_id, dirname)` — 创建目录**
1. 分配新 inode，类型设为 `DIDIR`
2. 分配一个数据块作为目录内容块
3. 初始化目录内容：`.`（指向自己）和 `..`（指向父目录）
4. 父目录的链接计数 `di_number++`
5. 在当前目录中添加新目录项

**`rmdir(user_id, dirname)` — 删除目录**
1. 查找目录 inode
2. 检查是否为空（只有 `.` 和 `..`）
3. 拒绝删除当前目录和根目录
4. 释放目录的数据块和 inode
5. 从父目录删除目录项
6. 父目录链接计数 `--`

**`dir()` — 列出当前目录**
- 遍历 `dir.direct` 数组，格式化输出每个目录项的信息（文件名、inode、类型、大小、权限）

### 3.12 name.c（路径解析）

**`namei(name)` — 文件名 → inode 号**
1. 在当前目录的 `dir.direct` 数组中遍历查找
2. 匹配 `d_name`，返回 `d_ino`
3. 未找到返回 0

**`iname(inodeid, name)` — inode 号 → 文件名**
- 反向查找，遍历目录项匹配 `d_ino`

### 3.13 access.c（权限检查）

**`access(user_id, inode, mode)` — 权限验证**
1. 如果用户是 root（uid=0），直接通过
2. 比较 inode 的 `di_uid` 与用户的 uid
3. 根据用户身份（所有者/同组/其他）取对应的 rwx 权限位
4. 检查请求的权限（READ/WRITE/EXECUTE）是否被允许

### 3.14 chmod.c / chown.c（权限/所有者修改）

**`chmod(user_id, filename, mode)`**
1. `namei` 查找文件
2. 检查是否为所有者
3. 保留高位的文件类型标志，修改低 9 位权限
4. `iput` 写回

**`chown_file(user_id, filename, new_uid)`**
1. `namei` 查找文件
2. 检查是否为所有者
3. 修改 `di_uid`
4. `iput` 写回

### 3.15 link.c（硬链接与软链接）

**`ln(user_id, src, dst)` — 硬链接**
1. 查找源文件 inode
2. 在当前目录添加新目录项，指向同一个 inode
3. `di_number++`（链接计数增加）
4. `iput` 写回

**`symlink(user_id, src, dst)` — 软链接**
1. 创建一个新文件，类型为 `DILINK`
2. 文件内容存储目标路径字符串

### 3.16 mv.c（重命名/移动）

**`mv(user_id, src, dst)` — 重命名**
1. 查找源文件 inode
2. 如果目标已存在，先删除目标
3. 在当前目录中修改源目录项的 `d_name` 为目标名
4. 如果跨越目录（本项目当前只支持同目录重命名）

### 3.17 lseek.c（文件指针定位）

**`lseek_file(user_id, fd_num, offset)`**
1. 验证 fd 有效性
2. 将文件偏移量设为指定的 offset
3. 注意：本项目实现的 `lseek` 是绝对定位，不是标准 C 的相对定位

### 3.18 stat.c / wc.c / find.c / df.c（工具命令）

**`stat(user_id, filename)`**
- 输出 inode 的完整元数据：ino、mode、uid、gid、size、links、blocks、ctime

**`wc(user_id, filename)`**
- 统计文件的行数、单词数、字节数

**`find(user_id, path, pattern)`**
- 递归遍历目录树，匹配文件名

**`df()`**
- 输出磁盘空间统计：总块数、已用块数、空闲块数、总 inode、已用 inode、空闲 inode

### 3.19 format.c（格式化）

**`format()` — 初始化文件系统**
1. 创建 `disk.img` 文件，填充 0
2. 初始化根目录 inode（ino=1）
3. 初始化 `etc` 目录 inode（ino=2）
4. 初始化密码文件 inode（ino=3）
5. 调用 `sb_init_format()` 初始化超级块和成组链接法
6. 写入根目录内容（`.` 和 `..`）
7. 写入 `etc` 目录和默认密码

### 3.20 install.c（文件系统挂载）

**`install()` — 加载已有文件系统**
1. 打开 `disk.img`
2. `sb_read()` 读取超级块
3. 初始化 inode Hash 表
4. 初始化系统打开文件表
5. 初始化用户表
6. 读取密码文件到内存
7. `iget(1)` 读取根目录 inode，加载根目录内容到 `dir`

### 3.21 halt.c（退出保存）

**`halt()` — 安全退出**
1. 将内存中的当前目录 `dir` 写回磁盘（分配新块或复用旧块）
2. 关闭所有用户打开的文件
3. `sb_write()` 写回超级块
4. `fclose(fd)` 关闭磁盘镜像

### 3.22 superblock.c / superblock.h（超级块管理）

**`sb_init_format()`**
- 格式化时初始化超级块，建立成组链接法的组长块链

**`sb_read()` / `sb_write()`**
- 从磁盘 1# 块读写超级块

**`sb_sync()`**
- 如果超级块被标记为脏（`s_fmod & SUPDATE`），写回磁盘

### 3.23 log.c（日志）

（当前版本基本为空或只有简单输出，尚未实现 WAL）

### 4.2 fs_client.py（TCP 客户端）

**`FsClient` 类功能：**

| 方法 | 功能 |
|------|------|

## 五、前后端通信协议

### 5.1 连接模型

- 后端启动 `./fs --daemon --tcp --host 127.0.0.1 --port <动态端口>`
- 后端使用**单连接模型**：一次只处理一个 TCP 客户端
- 前端通过绑定到端口 0 自动寻找本地空闲端口，避免端口冲突
- 前端每次发送命令都新建一个 TCP 连接（`send_command`）
- `send_batch` 在单个连接中顺序发送多条命令，保持 fd 状态

### 5.2 命令格式

前端 → 后端：命令字符串 + `\n`
后端 → 前端：命令输出 + `__END__\n`

例如：
```
前端发送: "dir\n"
后端返回: "CURRENT DIRECTORY\n...\n__END__\n"
```

### 5.3 内部查询命令

| 命令 | 输出格式 | 用途 |
|------|----------|------|
| `__disk_map__` | `0 0 1 0 1 ...`（512 个 0/1） | 磁盘块占用状态 |
| `__inode_list__` | `ino\|type\|size\|mode\|...` | Inode 列表 |
| `__dir_list__` | `name\|ino\|type\|size\|mode` | 当前目录文件表 |
| `__tree__` | 缩进文本树 | 目录树视图 |
| `__stats__` | `key=value\n...` | 系统统计 |
| `__block_info__` | `block\|ino\|role\|index` | 块归属映射 |
| `__snapshot__` | `__SECTION__ name\n...` | 一次性返回上述 6 段数据，减少 socket 往返 |

**`__snapshot__` 输出格式示例：**
```
__SECTION__ disk
0 0 1 0 0 ...
__SECTION__ inodes
ino|type|size|mode|...
...|...|...|...
__SECTION__ dir
name|ino|type|size|mode
...|...|...|...
__SECTION__ tree
/
  etc/
__SECTION__ stats
total_blocks=512
...
__SECTION__ blocks
block|ino|role|index
...|...|...|...
```

内部命令无 ANSI 颜色码，输出原始数据。用户命令（如 `dir`、`cat`）带有 ANSI 颜色，前端用正则 `\x1b\[[0-9;]*m` 过滤。

### 5.4 Daemon 生命周期

```
前端启动
  → start_daemon(): Popen([./fs, --daemon, --tcp, --host, 127.0.0.1, --port, <动态端口>])
  → _find_free_port(): 绑定到 127.0.0.1:0 获取空闲端口
  → 轮询 _can_connect() 检查该端口
  → 端口就绪，进入主循环
  → 用户关闭窗口
  → closeEvent()
  → stop_daemon(): SIGTERM → wait(2s) → SIGKILL
```

如果前端异常退出（如 `^C`），daemon 可能残留，需要手动 `pkill -f './fs --daemon'` 或运行 `run.sh` 自动清理。

## 七、设计决策与权衡

### 7.1 为什么不用 mmap？

`disk.img` 只有约 280KB，完全可以 mmap 到内存中。但本项目是教学用的文件系统模拟器，目的是展示文件系统如何通过块 I/O 管理磁盘。使用 `fseek`/`fread`/`fwrite` 更贴近真实文件系统的实现。

### 7.2 为什么不用 Buffer Cache？

当前版本每次 I/O 都直接操作 `disk.img`，没有块缓存层。这确实是性能瓶颈，但考虑到：
1. 磁盘镜像很小（280KB），现代 OS 的文件系统缓存已经能很好地缓存整个文件
2. 教学目的下，简单的直接 I/O 更容易理解
3. 如果自行实现 Buffer Cache，需要修改几乎所有有 I/O 的模块，工程量较大

### 7.3 为什么用单连接模型？

后端 `accept()` 循环一次只处理一个客户端。这是因为：
1. 文件系统的 fd 表、当前目录等状态是进程级的，多连接并发会导致状态混乱
2. 教学场景下，通常只有一个用户操作
3. 前端通过快速新建/关闭连接来实现并发效果

### 7.4 为什么用 delete+recreate 实现文件编辑？

后端没有 `edit` 或 `truncate` 命令。前端通过 `delete` 删除旧文件，再用 `create` 创建同名文件并写入新内容，来实现编辑保存。这简化了后端逻辑，但有一个副作用：如果文件有硬链接，编辑后会打断链接关系（新文件的 inode 号不同）。
