## 1 概述

操作系统作为计算机系统中最核心的系统软件，其文件管理子系统负责对外存空间进行组织、分配、回收与保护，直接决定了数据存储的效率、可靠性与安全性。在本科阶段，《操作系统原理》课程通常以理论讲授为主，学生虽然能够掌握文件系统的基本概念，如 inode、目录项、索引分配、空闲空间管理等，但缺乏将这些概念转化为具体代码实现的机会。操作系统课程设计正是弥补这一不足的关键实践环节。

本次课程设计的选题为“多用户、多级目录结构文件系统的设计与实现”，属于教学大纲中明确列出的核心设计题目之一。该题目要求学生从零开始设计一个具备实际可用性的文件系统，涵盖磁盘格式化、inode 分配与回收、数据块分配与回收、目录管理、文件读写、权限控制、用户管理、可视化界面等多个子系统。其难度与工作量均高于普通的课程实验，需要学生综合运用操作系统原理、数据结构与算法、C 语言程序设计、计算机网络与图形界面开发等多门课程的知识。

本设计的目标是构建一个**类 UNIX 的模拟文件系统**，它以普通文件 `disk.img` 作为虚拟磁盘，在逻辑上完全模拟真实磁盘的分块结构；后端采用标准 C 语言编写，不依赖任何第三方库，以保证代码的可移植性与教学可读性；前端采用 Python + PyQt6 实现跨平台 GUI，通过 TCP Socket 与后端 Daemon 通信，将文件系统内部的运行状态实时可视化。该系统不仅支持常规的创建、读写、删除、复制、移动文件等操作，还支持多级目录、硬链接、软链接、大文件一级间接索引、成组链接法、UGO 权限模型、root 特权用户等高级特性。

在后端架构上，我们采用了**模块化设计**思想，将文件系统划分为若干相对独立、接口清晰的子模块：超级块管理（`superblock.c`）、inode 缓存与分配（`igetput.c`、`iallfre.c`）、数据块分配（`ballfre.c`）、文件读写与多级索引（`rdwt.c`）、文件创建打开关闭（`creat.c`、`open.c`、`close.c`）、目录操作（`dir.c`、`name.c`）、权限检查（`access.c`）、属性修改（`chmod.c`、`chown.c`）、链接（`link.c`）、移动（`mv.c`）、工具命令（`stat.c`、`wc.c`、`find.c`）、格式化与安装（`format.c`、`install.c`）、主控与 Daemon（`main.c`）。每个模块对应一个或一对源文件，模块之间通过 `filesys.h` 中声明的数据结构与函数接口进行交互，极大地降低了系统的耦合度，也便于调试与功能扩展。

在前端设计上，我们充分利用 PyQt6 的丰富组件，构建了包含**磁盘块占用网格**、**目录树视图**、**当前目录列表**、**inode 元数据表**、**系统统计面板**与**命令终端**六大区域的综合界面。前端不仅能够发送常规命令，还能通过内部查询命令（如 `__snapshot__`）一次性获取文件系统的全部状态数据，实现面板的秒级刷新。此外，前端还集成了**自动测试**与**逐条测试**功能，能够按照预设脚本自动执行命令序列，并在每一步刷新面板，为教学演示与功能验收提供了极大便利。

本报告将从课程设计任务与要求出发，系统地阐述文件系统的算法与数据结构、程序设计与实现细节、实验测试结果，最后总结设计过程中的收获与体会。

## 2 课程设计任务及要求

### 2.1 设计任务

根据《操作系统课程设计》教学大纲，本次设计的核心任务是：**在深入理解操作系统文件管理子系统基本原理的基础上，设计并实现一个多用户、多级目录结构的模拟文件系统**。具体任务可细分为以下几个方面：

1. **虚拟磁盘设计与格式化**：确定磁盘块大小、inode 数量、数据块数量、各区域在磁盘上的物理布局；实现格式化程序，能够初始化超级块、建立根目录、建立用户口令表，并将成组链接法的初始空闲块链写入磁盘。

2. **核心数据结构实现**：定义并实现磁盘 inode（`dinode`）、内存 inode（`inode`）、目录项（`direct`）、超级块（`filsys`）、打开文件表（`file`）、用户表（`user`）、口令表（`pwd`）等关键数据结构；确保结构体在磁盘上的二进制布局精确可控，避免因编译器填充导致的磁盘映射错误。

3. **inode 与数据块管理**：实现 inode 的分配（`ialloc`）与回收（`ifree`），采用超级块缓存栈加速分配；实现数据块的分配（`balloc`）与回收（`bfree`），采用成组链接法管理空闲数据块，确保分配与回收均为 O(1) 时间复杂度。

4. **文件读写与索引机制**：实现文件的按字节读写（`read`、`write`），支持跨块读写；采用混合索引分配策略（直接索引 + 一级间接索引），使得小文件访问高效、大文件得以支持。

5. **目录与路径管理**：实现目录的创建（`mkdir`）、删除（`rmdir`）、切换（`cd`）、列出（`dir`）等功能；维护当前目录的内存缓存，并在目录切换时正确写回旧目录、读入新目录；支持 `.` 与 `..` 特殊目录项。

6. **权限控制与用户管理**：实现基于 uid/gid/mode 的标准 UNIX 权限模型，包含 owner、group、others 三级权限划分；引入 root 特权用户（uid=0），使其能够绕过普通权限检查；实现用户登录（`login`）、注销（`logout`）与口令验证。

7. **链接与属性变更**：实现硬链接（`ln`）与软链接（`symlink`），正确处理链接计数与链接文件内容；实现文件权限修改（`chmod`）与所有者变更（`chown`）。

8. **工具命令**：实现 `stat`（查看 inode 元数据）、`wc`（统计行/词/字节）、`find`（递归查找）、`df`（磁盘空间使用）、`lseek`（文件指针定位）等辅助命令。

9. **可视化运行界面**：设计并实现图形化前端，能够实时展示磁盘块占用状态、inode 分配情况、目录树、当前目录内容、系统统计信息；前端与后端通过 TCP 协议通信，支持命令发送与结果回显。

10. **测试与验收**：制定全面的测试方案，设计覆盖常规操作、边界条件、权限冲突、错误处理、持久化恢复、格式化重置等场景的测试用例；运行测试并通过验收。

## 3 算法与数据结构

### 3.1 总体设计思想与系统架构

本文件系统的设计深受经典 UNIX V6/V7 文件系统的启发，采用了**inode + 目录项 + 混合索引 + 成组链接法**的经典架构。总体设计思想可以概括为：

- **以 inode 为中心**：所有文件与目录的元数据统一由 inode 管理，目录项仅保存“文件名 → inode 编号”的映射关系。这种解耦使得文件重命名、硬链接等操作极为高效——只需修改目录项，无需移动文件数据。
- **小文件优先**：绝大多数实际文件都很小，因此系统提供 9 个直接索引项，使得小文件无需额外的间接索引块即可直接定位所有数据块，最大化 I/O 效率。
- **大文件支持**：当文件超过 9 个数据块时，第 10 个地址项（`di_addr[9]`）指向一个一级间接索引块，该块可存储 256 个数据块号，将单个文件的最大容量扩展至约 135KB。
- **批量 I/O 与缓存**：成组链接法将空闲块号批量缓存在超级块中，避免了每次分配都读盘；内存 inode Hash 表缓存最近访问的 inode，避免重复的磁盘 I/O。
- **权限分级**：严格遵循 UNIX 的 UGO（User-Group-Others）权限模型，并引入 root 特权，既保证了多用户环境下的数据隔离，又保留了系统管理的灵活性。

系统整体架构分为**后端（C 语言）**与**前端（Python + PyQt6）**两层：

- **后端**：负责所有文件系统核心逻辑。启动时可选择命令行交互模式或 TCP Daemon 模式。在 Daemon 模式下，后端监听本地 TCP 端口，接受前端发来的命令字符串，执行后将输出通过 Socket 回传。
- **前端**：启动时自动启动后端 Daemon（动态分配端口以避免冲突），然后通过 TCP 连接发送命令。前端包含六大可视化面板，并支持命令终端、自动测试、逐条测试、历史状态回放等功能。

### 3.2 虚拟磁盘布局与超级块管理模块

#### 3.2.1 功能

虚拟磁盘布局与超级块管理模块负责定义整个文件系统在磁盘镜像文件上的物理组织结构，并在格式化时初始化这些结构，在运行时装载与同步超级块状态。它是文件系统的“元数据管理层”，所有其他模块都依赖它来确定 inode 区、数据区的起始位置以及空闲资源的分布情况。

#### 3.2.2 数据结构

磁盘镜像 `disk.img` 按 512 字节/块划分，物理布局如下：

| 区域 | 物理块号 | 块数 | 总字节数 | 说明 |
|------|---------|------|---------|------|
| 保留块 | 0 | 1 | 512B | 系统保留，可用于引导信息 |
| 超级块 | 1 | 1 | 512B | 文件系统全局元数据，对应 `struct filsys` |
| inode 区 | 2–33 | 32 | 16KB | 512 个磁盘 inode，每个 32 字节 |
| 数据区 | 34–545 | 512 | 256KB | 用户数据块，包含目录内容、文件内容、间接索引块 |

关键宏定义（来自 `filesys.h`）：

```c
#define BLOCKSIZ    512     /* 每个磁盘块 512 字节 */
#define DINODEBLK   32      /* inode 区占用 32 个块 */
#define FILEBLK     512     /* 数据区共 512 个数据块 */
#define DINODESIZ   32      /* 每个磁盘 inode 固定 32 字节 */
#define DINODESTART 2*BLOCKSIZ   /* inode 区在文件中的字节偏移 */
#define DATASTART   (2+DINODEBLK)*BLOCKSIZ  /* 数据区在文件中的字节偏移 */
```

**超级块结构 `struct filsys`** 定义了文件系统的全局状态：

```c
struct filsys{
    uint16_t s_isize;          /* inode 区占用的块数（固定 32） */
    uint32_t s_fsize;          /* 数据区总块数（固定 512） */
    uint16_t s_nfree;          /* 当前空闲数据块总数 */
    uint16_t s_pfree;          /* s_free 栈的当前栈顶指针 */
    uint16_t s_free[NICFREE];  /* 成组链接法缓存的空闲块号栈，NICFREE=50 */
    uint16_t s_ninode;         /* 当前空闲 inode 总数 */
    uint16_t s_pinode;         /* s_inode 栈中当前数量 */
    uint16_t s_inode[NICINOD]; /* 空闲 inode 号缓存栈，NICINOD=50 */
    uint16_t s_rinode;         /* 下次扫描空闲 inode 的起点 */
    char     s_fmod;           /* 超级块脏标记（SUPDATE） */
};
```

#### 3.2.3 算法

**格式化算法（`format.c` + `superblock.c`）**：

1. 创建或清空 `disk.img`，按总大小填充 0。
2. 初始化根目录 inode（ino=1）：类型 `DIDIR`，链接数 1，分配数据块 0，写入 `.` 和 `..` 目录项。
3. 初始化 `/etc` 目录 inode（ino=2）：类型 `DIDIR`，链接数 1，分配数据块 1。
4. 初始化 `/etc/password` 文件 inode（ino=3）：类型 `DIFILE`，分配数据块 2，写入默认用户口令表（含 root 用户 uid=0 和普通用户）。
5. 调用 `sb_init_format()` 建立成组链接法的初始空闲块链：
   - 数据块 3–511 共 509 个块为空闲。
   - 将 511, 510, ..., 462（前 50 个）直接放入超级块 `s_free[]` 栈，栈顶指针 `s_pfree = 0`。
   - 从第 50 个块开始，每 50 个为一组，最后一个块作为组长块，其内容保存下一组的块号数量与块号序列。
   - 最后一组可能不足 50 个，按实际数量写入组长块。
6. 初始化空闲 inode 栈：除 ino=0（保留）、ino=1（根目录）、ino=2（/etc）、ino=3（password）外，其余 508 个 inode 号压入 `s_inode[]` 栈。
7. 将超级块写回磁盘块 1。

**安装算法（`install.c`）**：

1. 打开已有的 `disk.img`。
2. 调用 `sb_read()` 将磁盘块 1 读入内存全局变量 `filsys`。
3. 初始化 inode Hash 表 `hinode[128]`、系统打开文件表 `sys_ofile[40]`、用户表 `user[10]`。
4. 读取 `/etc/password` 到内存口令表 `pwd[32]`。
5. 调用 `iget(1)` 读取根目录 inode，并将其目录内容读入内存缓存 `dir`。
6. 设置当前路径为 `/`，等待用户命令。

**超级块同步算法**：

- 任何修改了 `filsys` 的操作（如 `balloc`、`bfree`、`ialloc`、`ifree`）都会设置 `s_fmod = SUPDATE`。
- 在 `halt()` 安全退出时，若 `s_fmod` 被置位，则调用 `sb_write()` 将内存超级块写回磁盘块 1，确保元数据持久化。

### 3.3 inode 管理模块

inode（Index Node，索引节点）是 UNIX 文件系统的核心抽象。本模块负责 inode 的内存缓存、磁盘读写、分配与回收。

#### 3.3.1 功能

- **内存缓存**：通过 Hash 链表缓存最近访问的 inode，避免重复的磁盘 I/O。
- **引用计数管理**：跟踪内存中各 inode 的被引用次数，当引用降为 0 时决定写回磁盘或回收资源。
- **分配与回收**：为新创建的文件/目录分配空闲 inode；当文件硬链接数降为 0 时回收 inode 及其数据块。

#### 3.3.2 数据结构

**磁盘 inode `struct dinode`**（精确 32 字节，写入磁盘）：

```c
struct dinode{
    uint16_t di_number;        /* 硬链接计数 */
    uint16_t di_mode;          /* 文件类型 + 权限 */
    uint16_t di_uid;           /* 所有者用户 ID */
    uint16_t di_gid;           /* 所属组 ID */
    uint32_t di_size;          /* 文件大小（字节） */
    uint16_t di_addr[NADDR];   /* 数据块索引数组，NADDR=10 */
};
```

**内存 inode `struct inode`**（扩展了内存管理字段）：

```c
struct inode{
    struct inode *i_forw;      /* Hash 链表后继指针 */
    struct inode *i_back;      /* 兼容字段 */
    char i_flag;               /* 脏标记（IUPDATE） */
    uint16_t i_ino;            /* inode 编号 */
    uint16_t i_count;          /* 内存引用计数 */
    uint16_t di_number;        /* 以下字段与 dinode 一一对应 */
    uint16_t di_mode;
    uint16_t di_uid;
    uint16_t di_gid;
    uint32_t di_size;
    uint16_t di_addr[NADDR];
};
```

**Hash 表**：`struct hinode hinode[NHINO]`，`NHINO=128`。每个桶是一个单向链表头，通过 `inodeid % 128` 定位桶。

#### 3.3.3 算法

**`iget(uint16_t dinodeid)` — 获取 inode**：

1. 计算 Hash 桶索引：`inodeid % NHINO`。
2. 遍历该桶的 Hash 链表，若找到 `i_ino == dinodeid` 的节点，则 `i_count++` 并返回该节点。
3. 若未命中，则 `malloc` 分配新的内存 inode，计算磁盘地址 `DINODESTART + dinodeid * DINODESIZ`，通过 `fseek`/`fread` 读取 32 字节磁盘 inode 内容到新节点的 `di_number` 起始位置（利用内存布局与磁盘布局的一致性）。
4. 初始化 `i_count = 1`、`i_flag = 0`、`i_ino = dinodeid`。
5. 将新节点插入 Hash 链表头部（头插法），以便下次快速命中。
6. 返回新节点。

**`iput(struct inode *pinode)` — 释放 inode**：

1. 若 `pinode` 为 NULL，直接返回。
2. 若 `pinode->i_count > 1`，仅 `i_count--` 后返回（仍有其他引用者）。
3. 若 `i_count == 1`，当前为最后一个引用：
   - 若 `di_number != 0`（仍有硬链接），将 inode 元数据写回磁盘（`fseek` + `fwrite`），保留文件。
   - 若 `di_number == 0` 且 `di_mode != DIEMPTY`（无硬链接），调用 `free_all_blocks()` 释放其占用的所有数据块，将 `di_mode` 置为 `DIEMPTY`、`di_size` 置为 0，写回磁盘，然后调用 `ifree()` 回收 inode 号。
4. 从 Hash 链表中摘除该节点，释放内存。

**`ialloc()` — 分配 inode**：

1. 若超级块空闲 inode 栈 `s_pinode > 0`，直接弹出栈顶 inode 号。
2. 若栈空，从磁盘 inode 区 `s_rinode` 开始向后扫描，寻找 `di_mode == DIEMPTY` 的空闲 inode，最多补充 `NICINOD=50` 个到栈中，更新 `s_rinode` 为扫描终止位置。
3. 若栈仍为空，说明 inode 耗尽，返回 NULL。
4. 弹出栈顶 inode 号后，调用 `iget()` 获取内存 inode，设置 `i_flag |= IUPDATE`，并将其初始状态同步写回磁盘。
5. 返回内存 inode 指针。

**`ifree(uint16_t dinodeid)` — 回收 inode**：

1. 边界检查：`dinodeid` 不得超出最大 inode 数；`s_ninode` 不得超出上限（防止重复释放导致计数错误）。
2. 查重：遍历当前 `s_inode[]` 栈，若该 inode 号已存在则直接返回。
3. 若栈未满（`s_pinode < NICINOD`），将 `dinodeid` 压入栈顶；否则仅更新 `s_rinode`（若 `dinodeid < s_rinode`，则降低扫描起点，使得下次 `ialloc` 能优先找到它）。
4. `s_ninode++`，`s_fmod = SUPDATE`。

### 3.4 目录项与路径解析模块

#### 3.4.1 功能

目录是文件系统的命名空间骨架。本模块负责目录项的查找、添加、删除，以及当前目录的缓存管理与切换。

#### 3.4.2 数据结构

**目录项 `struct direct`**（精确 16 字节）：

```c
struct direct{
    char d_name[DIRSIZ];       /* 文件名，DIRSIZ=14 */
    uint16_t d_ino;            /* 对应的 inode 编号 */
};
```

**当前目录缓存 `struct dir`**：

```c
struct dir{
    struct direct direct[DIRNUM];  /* 最多缓存 128 个目录项 */
    int size;                       /* 当前有效目录项数量 */
};
```

一个 512B 的数据块可容纳 `512 / 16 = 32` 个目录项。根目录初始包含 `.`（ino=1）和 `..`（ino=1），`/etc` 目录初始包含 `.`（ino=2）和 `..`（ino=1）。

#### 3.4.3 算法

**`namei(const char *name)` — 文件名 → inode 号**：

1. 在当前目录缓存 `dir.direct[0..dir.size-1]` 中线性遍历。
2. 若 `d_name` 与 `name` 匹配（使用 `strcmp`），返回对应的 `d_ino`。
3. 若遍历结束仍未找到，返回 `DIRNUM`（作为“未找到”的哨兵值）。

**`iname(const char *name)` — 添加目录项并返回位置**：

1. 检查文件名是否已存在（调用 `namei`），若存在则报错返回。
2. 检查文件名合法性（长度、非法字符等）。
3. 在 `dir.direct[]` 中查找第一个空闲槽（`d_ino == 0`）。
4. 将 `name` 拷贝到 `d_name`，暂时不设置 `d_ino`（由调用者如 `mkdir`、`creat` 设置）。
5. 返回该槽的索引。

**`mkdir(const char *dirname)` — 创建目录**：

1. 调用 `namei` 检查目录名是否已存在，若存在则报错。
2. 调用 `iname` 获取父目录中的空闲目录项槽位 `dirpos`。
3. 调用 `ialloc()` 分配新 inode，设置其类型为 `DIDIR`、链接数 1、所有者与组为当前用户、权限为默认模式。
4. 调用 `balloc()` 分配一个数据块作为目录内容块，初始化内容为 `.`（指向自己）和 `..`（指向父目录）。
5. 将新目录项的 `d_ino` 设置为新 inode 号，父目录链接数加 1（因为子目录的 `..` 指向父目录）。
6. `iput` 写回新 inode。

**`chdir(const char *dirname)` — 切换目录**：

1. 若 `dirname` 为 `.`，直接返回成功。
2. 调用 `namei` 查找目标目录项，若未找到则报错。
3. `iget` 获取目标 inode，调用 `access` 检查当前用户是否有该目录的执行权限（`EXECUTE`），若无则拒绝。
4. 检查目标是否为目录类型（`DIDIR`），若不是则报错。
5. 调用 `writeback_dir()` 将当前内存中的目录缓存写回磁盘（释放旧数据块、分配新数据块、写入 `dir.direct`）。
6. `iput(cur_path_inode)` 释放旧目录 inode。
7. 将 `cur_path_inode` 指向新 inode。
8. 从磁盘读取新目录的所有数据块到 `dir.direct`，更新 `dir.size`。
9. 返回成功。

**`rmdir(const char *dirname)` — 删除空目录**：

1. 拒绝删除 `.` 和 `..`。
2. `namei` 查找目录，获取 inode。
3. 检查目标是否为目录，若不是则报错。
4. 读取目录的所有数据块，检查除 `.` 和 `..` 外是否还有其他有效目录项；若有，则报错“目录非空”。
5. 检查当前用户是否拥有该目录的写权限。
6. 释放该目录占用的所有数据块，链接数减 1，调用 `iput`（由于 `di_number` 变为 0，inode 与数据块将被回收）。
7. 在父目录中移除该目录项（`dir_entry_remove`）。

### 3.5 空闲数据块管理模块（成组链接法）

#### 3.5.1 功能

管理数据区 512 个数据块的分配与回收，是文件系统空间管理的核心。本模块采用成组链接法（Grouped Free Block Linking），这是 UNIX 系统的经典算法，兼顾了分配回收效率与磁盘空间利用率。

#### 3.5.2 数据结构

成组链接法的状态完全存储在超级块中：

```c
uint16_t s_nfree;          /* 空闲数据块总数 */
uint16_t s_pfree;          /* s_free 栈当前栈顶指针 */
uint16_t s_free[NICFREE];  /* 当前一组空闲块号，NICFREE=50 */
```

此外，模块内部维护一个静态缓存数组用于读写组长块：

```c
static uint16_t block_buf[BLOCKSIZ / sizeof(uint16_t)];  /* 256 个 uint16_t */
```

#### 3.5.3 算法

**核心思想**：将 512 个空闲数据块分成若干组，每组最多 50 个块号。超级块始终缓存当前组的所有块号；当当前组用完时，从磁盘上的“组长块”读取下一组块号；当回收导致超级块栈满时，将当前组写入新回收的块，使其成为新的组长块。

**`balloc()` — 分配数据块**：

1. 若 `s_nfree == 0`，输出“Disk Full”并返回 `DISKFULL`。
2. 从 `s_free[s_pfree]` 取出栈顶块号 `free_block`。
3. 有效性校验：`free_block` 必须在 3–511 范围内，否则判定磁盘损坏。
4. **栈底判断**：若 `s_pfree == NICFREE - 1`（当前组的最后一个块号是组长块）：
   - 将 `free_block` 对应的磁盘块内容读入 `block_buf`。
   - `block_buf[0]` 为下一组的块号数量 `next_count`。
   - 将 `block_buf[1..next_count]` 反向装入 `s_free[]`：
     `s_free[NICFREE - 1 - i] = block_buf[i + 1]`，其中 `i = 0..next_count-1`。
   - 更新 `s_pfree = NICFREE - next_count`。
   - 返回 `free_block`（组长块本身也被当作普通数据块分配出去）。
5. **非栈底**：`s_pfree++`，直接返回 `free_block`。
6. `s_nfree--`，`s_fmod = SUPDATE`。

**`bfree(uint16_t block_num)` — 回收数据块**：

1. 若 `block_num < 3`，直接返回（系统保留块 0#/1#/2# 不回收）。
2. **栈满判断**：若 `s_pfree == 0`（超级块栈已满 50 个）：
   - 将当前栈中除组长块外的 49 个块号写入被回收块 `block_num`：
     `block_buf[0] = NICFREE - 1`（计数 49）；
     `block_buf[i+1] = s_free[NICFREE - 1 - i]`，`i = 0..48`。
   - 将 `block_buf` 写回磁盘 `DATASTART + block_num * BLOCKSIZ`。
   - 重置 `s_pfree = NICFREE - 1`（新栈只有一个元素，即新组长块）。
3. **栈未满**：`s_pfree--`。
4. 将 `block_num` 放入 `s_free[s_pfree]`。
5. `s_nfree++`，`s_fmod = SUPDATE`。

**成组链接法的优势**：

- **分配速度**：绝大多数分配操作直接从内存栈弹出，O(1)。
- **批量 I/O**：仅当当前组用完时才需要读取一次磁盘组长块，将随机 I/O 转化为批量 I/O。
- **空间零浪费**：组长块在成为组长之前是普通数据块，成为组长后其内容被覆盖为组信息；不需要额外的位图或链表指针空间。

### 3.6 文件读写与多级索引模块

#### 3.6.1 功能

实现文件的按字节读写，支持跨块访问；通过混合索引（直接索引 + 一级间接索引）将逻辑块号映射为物理块号，支持小文件的高效访问与大文件的扩展存储。

#### 3.6.2 数据结构

逻辑块到物理块的映射关系由 inode 的 `di_addr[10]` 数组决定：

| 索引范围 | 类型 | 容量 |
|---------|------|------|
| `di_addr[0..8]` | 直接索引 | 9 × 512B = 4608B |
| `di_addr[9]` | 一级间接索引块 | 指向一个包含 256 个块号的索引块，256 × 512B = 131072B |
| **合计** | | **265 块 ≈ 135680B** |

#### 3.6.3 算法

**`get_block(inode, logical_block)` — 逻辑块号 → 物理块号**：

1. 若 `logical_block < 9`，直接返回 `inode->di_addr[logical_block]`。
2. 若 `logical_block < 9 + 256`，说明进入间接索引范围：
   - 读取 `inode->di_addr[9]` 指向的间接索引块到缓冲区 `buf`。
   - 返回 `buf[logical_block - 9]`。
3. 超出范围返回 0。

**`set_block(inode, logical_block, physical_block)` — 建立映射**：

1. 若 `logical_block < 9`，直接设置 `inode->di_addr[logical_block] = physical_block`，返回成功。
2. 若 `logical_block < 9 + 256`：
   - 若 `di_addr[9] == 0`（尚未分配间接索引块），调用 `balloc()` 分配一个间接块，清空后写入磁盘，并将块号存入 `di_addr[9]`。
   - 读取间接索引块到 `buf`。
   - 设置 `buf[logical_block - 9] = physical_block`。
   - 将 `buf` 写回磁盘。
   - 返回成功。
3. 超出范围返回失败（文件过大）。

**`read(fd_num, buf, size)` — 读文件**：

1. 通过用户文件描述符 `fd_num` 索引到系统打开文件表项，获取 `f_inode` 与当前偏移 `f_off`。
2. 检查打开方式是否包含 `FREAD`。
3. 若 `f_off >= inode->di_size`，已读到末尾，返回 0。
4. 若 `f_off + size > di_size`，截断 `size` 为剩余可读字节数。
5. **跨块循环读取**：
   - 计算当前偏移所在的逻辑块号 `block = off / BLOCKSIZ` 与块内偏移 `block_off = off % BLOCKSIZ`。
   - 调用 `get_block()` 获取物理块号。
   - 若物理块号为 0，说明文件存在空洞，跳出循环。
   - 计算本次可读字节数 `to_read = min(BLOCKSIZ - block_off, bytes_left)`。
   - `fseek` 到 `DATASTART + physical_block * BLOCKSIZ + block_off`，`fread` 读取 `to_read` 字节到 `buf`。
   - 更新指针、偏移、剩余字节数。
6. 更新系统打开文件表的 `f_off`。
7. 返回实际读取的字节数。

**`write(fd_num, buf, size)` — 写文件**：

1. 类似 `read`，获取 inode 与偏移，检查 `FWRITE` 权限。
2. **跨块循环写入**：
   - 计算逻辑块号与块内偏移。
   - 调用 `get_block()` 获取物理块号。
   - 若物理块号为 0（该逻辑块尚未分配），调用 `balloc()` 分配新块；若分配失败则报错“磁盘满”。
   - 调用 `set_block()` 建立逻辑块到物理块的映射；若失败则报错“文件过大”并退回已分配块。
   - `fseek` + `fwrite` 写入 `to_write` 字节。
   - 更新指针、偏移、剩余字节数。
3. 若写完后 `off > di_size`，更新 `di_size`。
4. 设置 inode 脏标记 `i_flag |= IUPDATE`。
5. 更新 `f_off`。
6. 返回实际写入的字节数。

**`free_all_blocks(inode)` — 释放 inode 的所有数据块**：

1. 计算文件实际占用的块数 `nblocks = (di_size + BLOCKSIZ - 1) / BLOCKSIZ`（向上取整）。
2. 遍历直接索引项 `di_addr[0..8]`，对其中非零块号调用 `bfree()`。
3. 若 `di_addr[9] != 0`：
   - 读取间接索引块内容。
   - 遍历其中所有有效块号，逐个 `bfree()`。
   - 最后 `bfree()` 间接索引块本身。
4. 清空 `di_addr[]` 数组。

### 3.7 文件创建、打开与关闭模块

#### 3.7.1 功能

提供文件的创建、打开、关闭接口，管理用户级文件描述符表与系统级打开文件表，确保文件在多用户环境下的正确访问。

#### 3.7.2 数据结构

**系统打开文件表**：`struct file sys_ofile[SYSOPENFILE]`，`SYSOPENFILE=40`。

```c
struct file{
    char f_flag;               /* 打开方式：FREAD / FWRITE / FAPPEND */
    uint16_t f_count;          /* 引用计数（多个用户 fd 可共享同一系统表项） */
    struct inode *f_inode;     /* 指向已打开文件的内存 inode */
    uint32_t f_off;            /* 当前读写偏移 */
};
```

**用户打开文件表**：每个登录用户在 `struct user` 中维护一个 `u_ofile[NOFILE]` 数组（`NOFILE=20`），其元素值为系统打开文件表的下标。

```c
struct user{
    uint16_t u_active;         /* 用户槽是否被占用 */
    uint16_t u_default_mode;   /* 新建文件默认权限 */
    uint16_t u_uid;
    uint16_t u_gid;
    uint16_t u_ofile[NOFILE];  /* 用户文件描述符 → 系统打开文件表下标 */
};
```

#### 3.7.3 算法

**`creat(uid, filename, mode)` — 创建文件**：

1. 调用 `namei()` 检查文件名是否已存在。
2. 若存在且为普通文件（`DIFILE`），则将其截断：释放所有旧数据块（`free_all_blocks`），重置 `di_size=0`、`di_addr[]` 清零（相当于清空内容）。
3. 若不存在，调用 `ialloc()` 分配新 inode；调用 `iname()` 在当前目录添加目录项。
4. 设置 inode 的文件类型为 `DIFILE`，权限为 `mode`，所有者与组为当前用户，链接数 `di_number=1`。
5. 在系统打开文件表中寻找一个空闲项（`f_count==0`），设置 `f_flag=FWRITE`、`f_inode=新inode`、`f_off=0`、`f_count=1`。
6. 在用户文件描述符表中寻找空闲槽，填入系统表项下标。
7. 返回用户文件描述符 `fd`（0–19 范围内的索引）。

**`aopen(uid, filename, mode)` — 打开文件**：

1. 调用 `namei()` 查找文件名，获取 inode 号。
2. `iget()` 获取内存 inode。
3. 调用 `access()` 检查当前用户对该文件的权限：
   - 若 mode 包含 `READ`，需有读权限；
   - 若 mode 包含 `WRITE` 或 `APPEND`，需有写权限。
4. 在系统打开文件表中分配空闲项，设置 `f_flag`、`f_inode`、`f_off`。
5. 若 mode 为 `APPEND`，将 `f_off` 设为 `inode->di_size`（追加到末尾）。
6. 在用户 fd 表中分配槽位，返回 fd。

**`close(uid, fd_num)` — 关闭文件**：

1. 检查 `fd_num` 有效性。
2. 获取系统打开文件表项下标 `sys_idx = user[uid].u_ofile[fd_num]`。
3. `sys_ofile[sys_idx].f_count--`。
4. 若 `f_count == 0`，释放系统打开文件表项（重置为 0）。
5. 调用 `iput()` 写回 inode（若引用计数降为 0，inode 将被同步到磁盘或回收）。
6. 清除用户 fd 表项 `user[uid].u_ofile[fd_num] = 0`。

### 3.8 权限控制与 root 特权模块

#### 3.8.1 功能

实现类 UNIX 的访问控制机制，防止未授权用户读取、修改或执行文件。同时引入 root 超级用户，使其能够执行系统管理操作。

#### 3.8.2 数据结构

权限相关的常量定义（来自 `filesys.h`）：

```c
#define UDIREAD     00400   /* owner read */
#define UDIWRITE    00200   /* owner write */
#define UDIEXECUTE  00100   /* owner execute */
#define GDIREAD     00040   /* group read */
#define GDIWRITE    00020   /* group write */
#define GDIEXECUTE  00010   /* group execute */
#define ODIREAD     00004   /* other read */
#define ODIWRITE    00002   /* other write */
#define ODIEXECUTE  00001   /* other execute */
```

`di_mode` 的高 4 位表示文件类型（`DIFILE=01000`、`DIDIR=02000`、`DILINK=04000`），低 9 位表示 rwxrwxrwx 权限。

#### 3.8.3 算法

**`access(uid, inode, mode)` — 权限检查**：

1. **root 特权**：若 `user[uid].u_uid == 0`，直接返回 1（允许）。root 特权仅绕过权限检查，不绕过“文件不存在、目录非空、磁盘满”等结构性错误。
2. **owner 匹配**：若当前用户 uid 等于 `inode->di_uid`，根据 `mode`（READ/WRITE/EXECUTE）检查 `di_mode` 中对应的 owner 权限位。
3. **group 匹配**：若 uid 不匹配但 gid 等于 `inode->di_gid`，检查 group 权限位。
4. **other**：否则检查 other 权限位。
5. 若对应权限位为 1，返回 1；否则返回 0。

**`chmod(uid, filename, mode)` — 修改权限**：

1. `namei` 查找文件，获取 inode。
2. 检查当前用户是否为文件所有者（或 root）。
3. 保留 `di_mode` 高位的文件类型标志，仅修改低 9 位权限为新的 `mode`。
4. `iput` 写回。

**`chown_file(uid, filename, new_uid)` — 修改所有者**：

1. `namei` 查找文件。
2. 检查当前用户是否为文件所有者（或 root）。
3. 修改 `di_uid = new_uid`。
4. `iput` 写回。

### 3.9 链接与文件属性变更模块

#### 3.9.1 功能

实现 UNIX 文件系统的两种重要链接机制——硬链接与软链接，以及文件的重命名/移动操作。

#### 3.9.2 数据结构

硬链接不引入新数据结构，仅通过增加 `di_number`（链接计数）并在目录中添加新的指向同一 inode 的目录项来实现。

软链接创建一个新的 inode，其类型为 `DILINK`，文件内容存储目标路径字符串。

#### 3.9.3 算法

**`hlink(src, dst)` — 硬链接**：

1. `namei(src)` 查找源文件 inode。
2. `iget` 获取源 inode，检查其类型不能是目录（防止目录硬链接导致环状目录树）。
3. `iname(dst)` 在当前目录为目标名分配目录项。
4. 设置新目录项的 `d_ino = 源 inode 号`。
5. 源 inode 的 `di_number++`。
6. `iput` 写回源 inode。

**`slink(target, name)` — 软链接**：

1. `iname(name)` 分配目录项。
2. `ialloc()` 分配新 inode，类型设为 `DILINK`。
3. 分配一个数据块，将 `target` 路径字符串写入该块。
4. 设置 `di_addr[0] = 数据块号`，`di_size = strlen(target)`。
5. `iput` 写回新 inode。

**`mv(src, dst)` — 重命名/移动**：

1. `namei(src)` 查找源目录项索引 `src_id`。
2. 若目标名 `dst` 已存在，先删除目标（`delete`）。
3. 将源目录项的 `d_name` 修改为 `dst`（同目录重命名）。
4. 当前实现支持同目录重命名；跨目录移动可通过组合 `cp` + `delete` 实现。

### 3.10 用户登录与会话管理模块

#### 3.10.1 功能

维护多用户登录状态，验证用户身份，管理当前会话的 uid/gid，为权限检查提供基础。

#### 3.10.2 数据结构

**口令表 `struct pwd`**（存储于 `/etc/password` 文件）：

```c
struct pwd{
    uint16_t p_uid;
    uint16_t p_gid;
    char password[PWDSIZ];   /* PWDSIZ=12 */
};
```

系统预置用户：

| uid | gid | password | 说明 |
|-----|-----|----------|------|
| 0 | 0 | root | root 特权用户 |
| 2116 | 3 | dddd | 普通用户 |
| 2117 | 3 | bbbb | 普通用户 |
| 2118 | 4 | abcd | 普通用户（默认演示用户） |
| 2119 | 4 | cccc | 普通用户 |
| 2220 | 5 | eeee | 普通用户 |

#### 3.10.3 算法

**`login(uid, passwd)` — 用户登录**：

1. 在用户表 `user[10]` 中寻找空闲槽位（`u_active == 0`）。
2. 在口令表 `pwd[32]` 中查找 `p_uid == uid` 的条目。
3. 比较输入的口令与存储的口令，若不一致则登录失败。
4. 初始化用户槽：`u_active = 1`，`u_uid = uid`，`u_gid = 对应gid`，`u_default_mode = DEFAULTMODE(00777)`，`u_ofile[]` 清零。
5. 设置全局 `user_id` 为该槽位索引。
6. 返回成功。

**`logout(uid)` — 用户注销**：

1. 遍历当前用户的 `u_ofile[]`，关闭所有打开的文件。
2. 重置用户槽 `u_active = 0`。
3. 全局 `user_id` 置为 -1，当前路径重置为 `/`。

### 3.11 可视化前端与前后端通信模块

#### 3.11.1 功能

构建图形化用户界面，将抽象的文件系统内部状态转化为直观的可视化信息；通过 TCP 协议与后端 Daemon 通信，实现命令发送、结果回显与面板刷新。

#### 3.11.2 数据结构

前端使用 PyQt6 构建，主要组件包括：

- `DiskGridWidget`（自定义 QWidget）：32×16 网格，显示 512 个数据块的占用状态（系统保留/已占用/空闲）。
- `QTableWidget`：当前目录列表与 inode 列表。
- `QPlainTextEdit`：目录树视图与命令输出终端。
- `QLabel`：系统统计信息与块详情提示。
- `QSplitter`：实现面板的自由拖拽布局。

前后端通信协议：

- 后端启动参数：`./fs --daemon --tcp --host 127.0.0.1 --port <动态端口>`。
- 前端 → 后端：命令字符串 + `\n`。
- 后端 → 前端：命令输出 + `__END__\n`。
- 内部查询命令（无 ANSI 颜色，供前端解析）：
  - `__disk_map__`：512 个 0/1 空格分隔。
  - `__inode_list__`：`ino|type|size|mode|uid|gid|links|blocks` 表格。
  - `__dir_list__`：`name|ino|type|size|mode` 表格。
  - `__tree__`：缩进目录树文本。
  - `__stats__`：`key=value` 格式统计。
  - `__block_info__`：`block|ino|role|index` 块归属映射。
  - `__snapshot__`：一次性返回上述全部数据，以 `__SECTION__ name` 分隔。

#### 3.11.3 算法

**前端启动流程**：

1. `FsClient._find_free_port()` 绑定到 `127.0.0.1:0`，由操作系统分配空闲端口。
2. `FsClient.start_daemon()` 使用 `subprocess.Popen` 启动后端，传入动态端口参数。
3. 轮询 `_can_connect()` 检查端口是否就绪（最多 5 秒）。
4. 进入主窗口事件循环。

**面板刷新算法（`refresh_all()`）**：

1. 调用 `client.send_command("__snapshot__")`。
2. 接收返回文本，按 `__SECTION__` 标签分割为字典。
3. 更新 `DiskGridWidget`：解析 `disk` 段的 512 个 0/1，0 为空闲（深蓝），1 为已占用（绿），前三块为系统保留（橙）。
4. 更新 `inode` 表格：解析 `inodes` 段，按列填充。
5. 更新 `dir` 表格：解析 `dir` 段，显示当前目录内容。
6. 更新 `tree` 面板：直接显示 `tree` 段文本。
7. 更新 `stats` 面板：格式化显示总块数、已用块数、空闲块数、inode 数、当前用户、当前路径等。
8. 保存块归属映射：`block_info` 段解析为字典，用于点击磁盘块时显示归属文件。

**自动测试算法（`run_auto_tests()`）**：

1. 预定义约 60 条测试命令序列（包括登录、目录操作、文件创建/读写/关闭、权限测试、链接测试、大文件测试、清理等）。
2. 使用 `send_batch()` 在单个 TCP 连接中顺序发送命令（保持 fd 状态）。
3. 每发送一条命令，等待并接收输出，追加到终端显示，延时 1.0 秒。
4. 调用 `refresh_all()` 刷新所有面板，延时 0.5 秒。
5. 全部执行完毕后，弹出提示框。

**逐条测试算法（`run_step_test()`）**：

1. 维护 `_test_cmds` 列表与当前索引 `_test_idx`。
2. 每次点击“逐条测试”按钮，执行 `_test_cmds[_test_idx]`。
3. 索引自增，下次点击执行下一条。
4. 每步均调用 `refresh_all()` 刷新面板，便于观察单条命令对文件系统状态的影响。

### 4.1 程序流程图

本节以文字流程图的形式，描述文件系统核心操作的执行流程。由于课程设计报告以文本为主，以下采用结构化伪代码与流程描述相结合的方式，清晰展示各模块的调用关系与分支判断。

#### 4.1.1 文件系统启动与安装流程

```
开始
│
├─ 检查命令行参数
│   ├─ 若包含 --daemon --tcp：进入 TCP Daemon 模式
│   └─ 否则：进入交互式命令行模式
│
├─ 调用 install() 安装文件系统
│   ├─ 打开 disk.img
│   ├─ sb_read() 读取超级块到内存 filsys
│   ├─ 初始化 hinode[128] Hash 表
│   ├─ 初始化 sys_ofile[40] 系统打开文件表
│   ├─ 初始化 user[10] 用户表
│   ├─ 读取 /etc/password 到内存 pwd[]
│   ├─ iget(1) 读取根目录 inode
│   ├─ 将根目录内容读入 dir 缓存
│   └─ 设置 current_path = "/"
│
├─ 若 Daemon 模式
│   ├─ 创建 TCP Socket，绑定 127.0.0.1:port
│   ├─ listen() 监听连接
│   └─ accept() 循环，每来一个连接调用 handle_client()
│       ├─ 从 socket 读取命令行
│       ├─ dup2 重定向 stdout 到临时文件
│       ├─ 调用 process_cmd() 执行命令
│       ├─ 将临时文件内容通过 socket 回传
│       └─ 追加 "__END__\n"
│
└─ 若交互模式
    └─ 循环读取 stdin → process_cmd() → 输出结果
        └─ 直到收到 exit / halt 命令
            └─ 调用 halt()
                ├─ writeback_dir() 写回当前目录
                ├─ 关闭所有打开文件
                ├─ sb_write() 若超级块脏则写回
                └─ fclose(fd) 关闭磁盘镜像
```

#### 4.1.2 文件创建流程（creat）

```
开始 creat(uid, filename, mode)
│
├─ 调用 namei(filename) 查找是否已存在
│   ├─ 若存在且为 DIFILE：
│   │   ├─ 调用 free_all_blocks() 释放旧数据块
│   │   ├─ 重置 di_size = 0，di_addr[] = 0
│   │   └─ 复用该 inode（截断模式）
│   └─ 若不存在：
│       ├─ 调用 ialloc() 分配新 inode
│       └─ 调用 iname(filename) 在当前目录添加目录项
│
├─ 设置 inode 元数据
│   ├─ di_mode = DIFILE | mode
│   ├─ di_uid = user[uid].u_uid
│   ├─ di_gid = user[uid].u_gid
│   ├─ di_number = 1
│   └─ i_flag |= IUPDATE
│
├─ 在 sys_ofile[] 中寻找 f_count == 0 的空闲项
│   ├─ 设置 f_flag = FWRITE
│   ├─ f_inode = 新 inode
│   ├─ f_off = 0
│   └─ f_count = 1
│
├─ 在 user[uid].u_ofile[] 中寻找空闲槽
│   └─ 填入 sys_ofile 下标
│
└─ 返回用户文件描述符 fd
```

#### 4.1.3 文件读流程（read）

```
开始 read(fd_num, buf, size)
│
├─ 检查 fd_num 有效性（< NOFILE 且 sys_idx < SYSOPENFILE）
├─ 检查打开方式是否包含 FREAD
├─ 获取 inode 与当前偏移 off
│
├─ 若 off >= di_size：返回 0（已到末尾）
├─ 若 off + size > di_size：size = di_size - off（截断）
│
├─ bytes_left = size
└─ 循环 while bytes_left > 0
    ├─ block = off / BLOCKSIZ
    ├─ block_off = off % BLOCKSIZ
    ├─ physical_block = get_block(inode, block)
    │   ├─ 若 block < 9：返回 di_addr[block]
    │   └─ 否则：读取间接索引块，返回对应项
    ├─ 若 physical_block == 0：break（空洞）
    ├─ to_read = min(BLOCKSIZ - block_off, bytes_left)
    ├─ fseek(DATASTART + physical_block * BLOCKSIZ + block_off)
    ├─ fread(to_read 字节到 buf)
    ├─ buf += to_read
    ├─ off += to_read
    └─ bytes_left -= to_read
│
├─ 更新 f_off = off
└─ 返回实际读取字节数 = size - bytes_left
```

#### 4.1.4 文件写流程（write）

```
开始 write(fd_num, buf, size)
│
├─ 检查 fd_num 有效性与 FWRITE 权限
├─ 获取 inode 与当前偏移 off
├─ bytes_left = size
└─ 循环 while bytes_left > 0
    ├─ block = off / BLOCKSIZ
    ├─ block_off = off % BLOCKSIZ
    ├─ physical_block = get_block(inode, block)
    │
    ├─ 若 physical_block == 0（未分配）
    │   ├─ physical_block = balloc()
    │   ├─ 若 DISKFULL：报错"Disk full"，break
    │   └─ set_block(inode, block, physical_block)
    │       ├─ 若失败（超过间接索引上限）：bfree(physical_block)，报错"File too large"，break
    │
    ├─ to_write = min(BLOCKSIZ - block_off, bytes_left)
    ├─ fseek(DATASTART + physical_block * BLOCKSIZ + block_off)
    ├─ fwrite(to_write 字节)
    ├─ buf += to_write
    ├─ off += to_write
    └─ bytes_left -= to_write
│
├─ 若 off > di_size：di_size = off
├─ inode->i_flag |= IUPDATE
├─ 更新 f_off = off
└─ 返回实际写入字节数
```

#### 4.1.5 成组链接法分配流程（balloc）

```
开始 balloc()
│
├─ 若 s_nfree == 0：输出"Disk Full"，返回 DISKFULL
├─ free_block = s_free[s_pfree]
├─ 有效性检查：free_block 必须在 [3, FILEBLK-1]
│   └─ 否则：标记磁盘损坏，返回 DISKFULL
│
├─ 若 s_pfree == NICFREE - 1（栈底，组长块）
│   ├─ fseek(DATASTART + free_block * BLOCKSIZ)
│   ├─ fread(block_buf, BLOCKSIZ)
│   ├─ next_count = block_buf[0]
│   ├─ 若 next_count 非法：s_nfree = 0，返回 free_block
│   ├─ for i = 0 to next_count-1
│   │   └─ s_free[NICFREE - 1 - i] = block_buf[i + 1]
│   └─ s_pfree = NICFREE - next_count
│
└─ 否则
    └─ s_pfree++
│
├─ s_nfree--
├─ s_fmod = SUPDATE
└─ 返回 free_block
```

#### 4.1.6 目录切换流程（chdir）

```
开始 chdir(dirname)
│
├─ 若 dirname == "."：返回 1
├─ dirid = namei(dirname)
├─ 若 dirid >= DIRNUM：报错"不存在"，返回 0
│
├─ inode = iget(dir.direct[dirid].d_ino)
├─ 调用 access(user_id, inode, EXECUTE) 检查执行权限
│   └─ 无权限：报错，iput(inode)，返回 0
├─ 检查 inode->di_mode & DIDIR
│   └─ 非目录：报错，iput(inode)，返回 0
│
├─ writeback_dir(cur_path_inode) 写回旧目录
├─ iput(cur_path_inode) 释放旧 inode
├─ cur_path_inode = inode
│
├─ 从磁盘读取新目录的所有数据块到 dir.direct
├─ dir.size = inode->di_size / sizeof(struct direct)
└─ 返回 1
```

#### 4.1.7 权限检查流程（access）

```
开始 access(uid, inode, mode)
│
├─ 若 user[uid].u_uid == 0：返回 1（root 特权）
│
├─ 若 user[uid].u_uid == inode->di_uid（owner）
│   ├─ mode == READ  ? 检查 UDIREAD
│   ├─ mode == WRITE ? 检查 UDIWRITE
│   └─ mode == EXEC  ? 检查 UDIEXECUTE
│   └─ 返回对应位是否存在
│
├─ 否则若 user[uid].u_gid == inode->di_gid（group）
│   └─ 检查 GDIREAD / GDIWRITE / GDIEXECUTE
│
└─ 否则（other）
    └─ 检查 ODIREAD / ODIWRITE / ODIEXECUTE
```

### 4.2 程序说明

本节对项目中各核心源文件的功能、关键函数与实现细节进行系统说明，以便读者理解程序的整体结构与模块划分。

#### 4.2.1 main.c — 主控程序与 TCP Daemon

`main.c` 是文件系统的入口文件，规模约 1400 行，承担命令解析、TCP Daemon、内部查询命令三大职责。

- **全局变量定义**：在此定义了 `hinode[]`、`dir`、`sys_ofile[]`、`filsys`、`pwd[]`、`user[]`、`fd`、`cur_path_inode`、`user_id`、`current_path` 等全部全局变量，确保其他源文件通过 `extern` 正确链接。
- **命令解析（`process_cmd`）**：使用 `sscanf` 提取命令关键字，通过 `if-else` 链分发给对应的 `cmd_*` 处理函数。所有需要登录的命令（如 `dir`、`mkdir`、`create`、`open` 等）均先调用 `require_login()` 检查，若未登录则提示“Please login first.”。
- **TCP Daemon（`run_tcp_daemon`）**：创建 `AF_INET` 流式 Socket，绑定到指定 host:port，进入 `accept()` 循环。每个连接由 `handle_client()` 处理，使用 `dup2` 将 stdout 重定向到 `tmpfile`，执行命令后将临时文件内容回传给前端，最后追加 `__END__\n` 作为结束标记。在 Linux/macOS 下，由于 `read`/`write`/`close` 等函数名与 POSIX 系统调用冲突，特使用 `syscall(SYS_read/write/close)` 直接调用底层系统调用，避免递归调用自身。
- **内部查询命令**：为前端刷新面板提供无颜色、易解析的数据输出。`cmd_internal_snapshot()` 将 `__disk_map__`、`__inode_list__`、`__dir_list__`、`__tree__`、`__stats__`、`__block_info__` 六段数据一次性返回，以 `__SECTION__ name` 分隔，显著减少了 Socket 往返次数，提升了前端刷新效率。
- **辅助函数**：`parse_mode()` 将字符串 "read"/"write"/"append" 转换为 `FREAD`/`FWRITE`/`FAPPEND`；`inode_type_name()` 将 `di_mode` 转换为 "dir"/"file"/"link" 字符串；`print_inode_blocks()` 递归打印 inode 的直接块与间接块链，用于 `dir` 命令和内部查询。

#### 4.2.2 filesys.h — 核心头文件

`filesys.h` 是整个项目的“宪法”，定义了所有数据结构、常量宏、全局变量 `extern` 声明与函数原型。

- **紧凑布局**：使用 `#pragma pack(push, 1)` 禁止编译器自动填充，确保 `struct dinode`、`struct direct`、`struct filsys` 等在磁盘上的二进制大小与预期完全一致。这是将结构体直接作为磁盘块内容读写的前提。
- **定宽整数**：统一使用 `<stdint.h>` 中的 `uint16_t`、`uint32_t`，避免 16 位/32 位/64 位平台差异导致的数据宽度不一致问题。
- **常量宏**：明确定义了块大小（512B）、inode 数（512）、数据块数（512）、文件名长度（14）、打开文件表容量（40）、Hash 桶数（128）、成组链接法栈深（50）等关键参数。
- **权限常量**：使用八进制清晰定义了 `UDIREAD`、`UDIWRITE`、`UDIEXECUTE` 等 9 个权限位，便于位运算检查。
- **函数声明**：补齐了所有函数的 ANSI C 原型声明，包括本设计新增的 `chmod`、`chown_file`、`hlink`、`slink`、`lseek_file`、`mv`、`file_stat`、`file_wc`、`file_find` 等扩展功能。

#### 4.2.3 superblock.c / superblock.h — 超级块管理

- **`sb_init_format()`**：格式化时初始化超级块。核心任务是建立成组链接法的初始空闲块链。算法细节：将 509 个空闲块（3–511）倒序排列，前 50 个放入超级块栈，之后每 50 个为一组，组内块号反向写入组长块，形成链式结构。
- **`sb_read()` / `sb_write()`**：使用 `fseek` 定位到磁盘块 1 的起始地址，分别进行 `fread` 与 `fwrite`。由于超级块大小小于 512 字节，剩余空间被忽略，不影响功能。
- **`sb_sync()`**：检查 `s_fmod & SUPDATE`，若超级块被修改则调用 `sb_write()` 写回磁盘。该函数在 `halt()` 时被调用。

#### 4.2.4 igetput.c — inode 内存缓存

- **`iget()`**：Hash 缓存的核心。计算 `dinodeid % NHINO` 定位桶，遍历链表查找命中。未命中时从磁盘读取 32 字节 `dinode` 内容到 `malloc` 分配的内存 inode，并头插到 Hash 链表中。其设计保证了同一 inode 在内存中只有一份拷贝，所有引用共享该对象，通过 `i_count` 管理生命周期。
- **`iput()`**：引用计数的释放逻辑。当 `i_count > 1` 时仅减计数；当 `i_count == 1` 时，若 `di_number != 0` 则写回磁盘，否则释放数据块并回收 inode。Hash 链表的摘除采用遍历前驱的方式处理，虽然时间复杂度为 O(n)，但由于 Hash 桶深度通常很浅，性能可接受。

#### 4.2.5 iallfre.c — inode 分配与回收

- **`ialloc()`**：超级块空闲 inode 栈为空时，从 `s_rinode` 开始扫描磁盘 inode 区，以 512 字节为单位批量读取（一次可读 16 个 dinode），将 `DIEMPTY` 项补充到栈中。`s_rinode` 的设计避免了每次扫描都从 0 开始，是一种简单而有效的增量扫描策略。
- **`ifree()`**：栈满时（`s_pinode == NICINOD`）不越界写入数组，而是利用 `s_rinode` 记录更小的空闲 inode 号，使得下次 `ialloc` 扫描时能够优先发现它。这种“软回退”策略避免了复杂的磁盘链表结构，同时保证了空闲 inode 不会永久丢失。

#### 4.2.6 ballfre.c — 成组链接法

- **`balloc()`**：栈底组长块的处理是本模块的难点。当 `s_pfree == NICFREE - 1` 时，取出的块号本身是组长块，需要读取其内容获取下一组块号，反向装入超级块栈，然后返回该组长块供数据使用。反向装入的设计是为了抵消 `bfree()` 中反向存储的操作，使得栈顶始终是逻辑上下一个要分配的块。
- **`bfree()`**：栈满时，将当前超级块栈（除被回收块自身外）的 49 个块号写入被回收块，使其成为新组长块。计数值为 49 而非 50，是因为新回收块本身将成为新组的组长，不计入组内块号列表。

#### 4.2.7 rdwt.c — 文件读写与多级索引

- **`get_block()` / `set_block()`**：这两个函数是多级索引的核心抽象。`get_block` 将逻辑块号翻译为物理块号；`set_block` 负责建立映射，并在首次使用间接索引时自动分配间接块、初始化内容。
- **`read()` / `write()`**：均采用跨块循环算法，按 `BLOCKSIZ` 为单位分块处理。`write` 在发现逻辑块未分配时自动调用 `balloc()` 与 `set_block()`，实现了文件的动态增长。`write` 还负责维护 `di_size` 与 `IUPDATE` 脏标记。
- **`free_all_blocks()`**：释放 inode 所有数据块时，必须先释放间接块指向的数据块，再释放间接块本身，顺序不可颠倒，否则将丢失间接块中的块号信息，导致数据块泄漏。

#### 4.2.8 dir.c — 目录操作

- **`_dir()`**：列出当前目录时，不仅显示文件名，还显示完整的 `rwxrwxrwx` 权限字符串、文件大小、类型标识（`<dir>`、`<link>`）以及直接块与间接块的块链信息。权限字符串通过位运算逐位判断生成。
- **`mkdir()`**：新目录的数据块中必须包含 `.` 和 `..` 两个标准目录项。`.` 指向自己，`..` 指向父目录。因此，创建子目录会导致父目录的链接数 `di_number` 加 1。
- **`chdir()`**：切换目录前必须调用 `writeback_dir()` 将当前内存中的目录缓存写回磁盘。`writeback_dir()` 的实现策略是“释放旧块、分配新块、写入内容”，虽然对于仅修改少量目录项的场景略显粗暴，但保证了简单性与正确性。
- **`rmdir()`**：删除目录前必须检查目录是否为空（除 `.` 和 `..` 外无其他有效目录项）。检查方法为读取目录的所有数据块，遍历目录项数组。此外，禁止删除 `.` 和 `..`，防止破坏目录树结构。

#### 4.2.9 access.c — 权限检查

- **`access()`**：严格按照“root → owner → group → other”的顺序判断，不叠加权限。例如，若当前用户是文件所有者，则仅检查 owner 权限位，即使 group 或 other 拥有更高权限也不予考虑。这与标准 UNIX 的行为一致。

#### 4.2.10 format.c — 格式化

- **`format()`**：初始化整个磁盘镜像。除了创建根目录、`/etc` 目录和 `/etc/password` 文件外，还负责调用 `sb_init_format()` 建立成组链接法链。所有预分配 inode（ino=1,2,3）的数据块（0,1,2）在 `bfree` 中被标记为系统保留（`block_num < 3` 不回收），因此这些块不会被文件系统重新分配。

#### 4.2.11 前端文件说明

- **`frontend/fs_client.py`**：`FsClient` 类管理后端 Daemon 的整个生命周期。`start_daemon()` 动态寻找空闲端口并启动后端进程；`stop_daemon()` 发送 SIGTERM，必要时 SIGKILL 强制终止；`send_command()` 建立 TCP 连接、发送命令、读取到 `__END__` 为止；`send_batch()` 在单个连接中批量发送命令，保持 fd 状态，用于自动测试；`refresh()` 发送 `__snapshot__` 并解析返回的六段数据。
- **`frontend/main.py`**：`MainWindow` 类构建完整的 GUI。使用 `QSplitter` 实现可拖拽布局；`DiskGridWidget` 自定义绘制 32×16 的块网格，支持点击与悬浮提示；表格组件使用 `configure_table()` 统一配置为只读、整行选择、隔行背景色、可排序；命令终端支持 ↑↓ 键浏览历史命令；回放控制条通过保存/恢复 `_snapshots` 列表实现历史状态查看。
- **`frontend/widgets/disk_grid.py`**：自定义 QWidget，重写 `paintEvent` 使用 `QPainter` 绘制网格。颜色编码：橙色（系统保留块 0–2）、绿色（已占用）、深蓝色（空闲）。`mousePressEvent` 计算点击位置对应的块索引，发射 `blockSelected(int)` 信号。

## 5 结论

本次操作系统课程设计围绕“多用户、多级目录结构文件系统”这一核心题目，从需求分析、架构设计、算法选型、编码实现到测试验收，完整地经历了一个中型系统软件的开发周期。通过本次设计，我们成功实现了一个功能完备、结构清晰、具有可视化界面的类 UNIX 模拟文件系统，主要取得了以下成果：

1. **核心功能完备**：系统支持文件与目录的创建、删除、读写、复制、移动、重命名；支持多级目录树、硬链接、软链接；支持基于 uid/gid/mode 的标准 UNIX 权限模型与 root 特权用户；支持大文件的一级间接索引；支持用户登录与会话管理。

2. **空间管理高效**：采用成组链接法管理空闲数据块，实现了 O(1) 的分配与回收速度，同时避免了位图或链表方案的空间或性能缺陷；采用超级块缓存栈管理空闲 inode，结合增量扫描策略，保证了分配效率。

3. **缓存与 I/O 优化**：通过内存 inode Hash 表缓存最近访问的 inode，减少了重复的磁盘读取；前后端通信引入 `__snapshot__` 批量查询机制，将面板刷新所需的 6 次 Socket 往返压缩为 1 次，显著提升了前端响应速度。

4. **可视化与教学价值突出**：PyQt6 前端实时展示磁盘块占用、inode 元数据、目录树、系统统计等信息，使抽象的文件系统内部机制变得直观可感；自动测试与逐条测试功能为教学演示与功能验证提供了极大便利；回放功能允许用户回溯历史状态，便于理解操作之间的因果关系。

5. **健壮性与可移植性良好**：后端使用标准 C 语言编写，不依赖第三方库，可在 macOS、Linux 乃至 Windows（通过 MinGW/MSYS2）上编译运行；前端使用 PyQt6，跨平台能力强。系统对各类非法输入与异常情况均有明确的错误提示，避免了崩溃与数据损坏。

当然，本系统作为教学用的模拟文件系统，与真实的生产级文件系统相比仍存在一定差距：

- **缺少 Buffer Cache**：当前每次 I/O 都直接操作 `disk.img`，没有实现块级缓冲层。虽然得益于现代操作系统的文件缓存，实际性能尚可，但从教学完整性角度，Buffer Cache 的实现将是后续扩展的重要方向。
- **单连接模型限制了并发**：后端 Daemon 一次只处理一个 TCP 连接，无法支持多客户端并发访问。这主要是为了避免进程级状态（如当前目录、打开文件表）的并发冲突。若需支持并发，需要引入会话隔离或更复杂的锁机制。
- **缺少日志与崩溃恢复**：当前系统仅在正常退出时同步超级块与目录缓存，若在运行中异常崩溃（如前端强制退出、系统断电），可能导致超级块与实际空闲块状态不一致。引入 Write-Ahead Logging（WAL）或简单的日志结构将是提高可靠性的关键。
- **路径解析较为简单**：当前仅支持单层文件名解析（`namei` 在当前目录中查找），不支持绝对路径与多级相对路径（如 `cd workdir/subdir`）。`find` 命令虽然支持递归遍历，但 `cd`、`open` 等命令仍依赖手动逐层进入。
- **软链接解析未完全实现**：当前软链接文件的内容仅为目标路径字符串，`cat` 命令直接显示该字符串而非解析并显示目标文件内容。完整的符号链接解析需要在 `open`、`cat`、`stat` 等命令中增加递归解析逻辑。

总体而言，本次课程设计达到了预期的教学目标，不仅加深了我们对操作系统文件管理子系统的理解，也锻炼了我们从系统设计到代码实现、从前端交互到网络通信的综合工程能力。文件系统作为操作系统中最复杂的子系统之一，其设计中的每一个决策（如 inode 与目录项的分离、混合索引、成组链接法、权限分级）都蕴含着深刻的工程智慧，值得我们反复揣摩与深入学习。

## 7 收获、体会和建议

### 7.1 收获

通过本次操作系统课程设计，我们在理论知识、工程实践与团队协作三个层面均获得了显著的提升。

**理论层面**：在课堂上学到的 inode、目录项、索引分配、空闲空间管理、访问控制等概念，曾经只是书本上的抽象定义。而通过亲手实现一个文件系统，我们真正理解了这些概念之间的有机联系。例如，inode 与目录项的分离设计，使得硬链接的实现变得异常简洁——只需增加一个目录项和一个链接计数即可，无需复制任何数据。又如，成组链接法的设计精妙之处在于它用“超级块缓存 + 磁盘组长块链”两级结构，将随机 I/O 转化为批量 I/O，这种在时空权衡上的智慧，若不亲手编码很难深刻体会。

**工程层面**：我们学会了如何将一个复杂的系统分解为若干高内聚、低耦合的模块；学会了如何使用版本控制（Git）管理代码、如何编写 Makefile 构建项目、如何编写 Shell 脚本自动化测试。更重要的是，我们深刻体会到了**磁盘数据一致性**的重要性。由于结构体直接映射到磁盘块，任何内存中的脏数据若未及时写回，都可能导致文件系统损坏。因此，我们在设计时刻意引入了 `IUPDATE` 和 `SUPDATE` 脏标记，并在关键路径（如 `iput`、`halt`）中确保同步，这种对数据一致性的敏感度将成为我们今后从事系统软件开发的重要素养。

**技术栈拓展**：后端使用纯 C 语言实现，加深了我们对指针、内存管理、位运算、文件 I/O 的掌握；前端使用 Python + PyQt6，让我们体验了从系统级编程到应用级 GUI 开发的全栈流程；前后端通过 TCP Socket 通信，使我们复习了网络编程中的 Socket 创建、绑定、监听、接受、读写、关闭等完整流程。

### 7.2 体会

本次设计中最深的体会是：**操作系统是细节的艺术**。文件系统看似只是“存取文件”，但其背后的每一个细节都可能成为致命的 bug。例如，在实现 `bfree()` 时，我们曾一度困惑“栈满时写入组长块的计数应该是 49 还是 50”。经过反复推导与测试，我们才确认计数应为 49，因为新回收的块本身将成为新组的组长，不应被计入组内的块号列表。若此处写为 50，则 `balloc()` 在读取时会多取一个无效的块号，导致磁盘损坏。又如，在 `iput()` 中，释放 inode 前必须先释放其数据块，再写回磁盘并回收 inode，顺序稍有颠倒就可能导致数据块泄漏（inode 已回收，但数据块仍被标记为占用）。这些细节让我们对操作系统代码的严谨性有了切身的敬畏。

另一个体会是**可视化对理解系统的重要性**。在仅有命令行输出的阶段，我们虽然能够验证功能正确性，但很难直观地感受“创建一个文件到底占用了哪些磁盘块”、“删除一个文件后哪些块被释放”。前端可视化面板的出现，使得这些抽象的内部状态一目了然。看到磁盘块网格中绿色块随命令执行而增减，看到 inode 列表中链接数随硬链接操作而变化，这种即时反馈极大地加深了我们对文件系统行为的理解。这也启发我们，在今后设计复杂系统时，应尽早考虑可观测性与可视化工具的建设。

### 7.3 建议

基于本次课程设计的经验，我们对课程设计教学提出以下建议：

1. **增加前置知识梳理环节**：文件系统涉及的知识点较多（inode、成组链接法、多级索引、权限位运算等），建议在课程设计开始前安排 2–3 学时的专题讲解，帮助学生梳理知识脉络，减少起步阶段的迷茫。

2. **提供渐进式代码框架**：完全从零开始实现文件系统对部分同学而言难度过大。建议指导教师提供一个最小可运行的代码框架（如已定义好数据结构、已实现磁盘读写工具函数），让学生在此基础上填充核心算法，既能保证教学重点，又能降低挫败感。

3. **强化测试驱动的理念**：建议在课程设计中明确要求学生编写自动化测试脚本，并将测试覆盖率作为评分参考之一。测试不仅是验证功能的手段，更是帮助学生在编码初期就思考边界条件与异常处理的重要方法。

4. **鼓励前端可视化创新**：可视化前端不仅能提升项目的展示效果，更能倒逼学生深入理解文件系统的内部状态。建议将“可视化运行界面”的评分权重适当提高，并鼓励学生在前端设计上进行创新（如块级依赖图、3D 磁盘模型、操作历史动画等）。

5. **组织代码评审与互评**：在课程设计中期组织小组间的代码评审或互评，让学生相互学习优秀的模块划分、算法实现与代码风格。这不仅能提高代码质量，也能培养学生的技术沟通与批判性思维能力。
